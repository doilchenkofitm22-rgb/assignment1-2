 # Процес DAST (динамічне тестування)

## Кроки

1. Запустити додаток локально: `docker-compose up -d` або `make up`, щоб отримати доступ до сервісу за адресою `http://localhost:5000`.
2. Запустити baseline‑сканування: `make zap` (скрипт у репозиторії) або дочекатись виконання workflow `dast.yml` у CI. Задаються URL, що тестуються: домашня сторінка, `/search?q=test`, `/login`, `/echo?msg=test`.
3. Після завершення тесту ZAP генерує HTML‑звіт, який зберігається в каталозі `reports/zap/` та завантажується як артефакт CI.

## Результати та висновки

У ході тестування ZAP виявив кілька проблем:

| Категорія | Опис | Критичність |
|----------|------|-------------|
| SQL Injection | ZAP виявив, що передача рядків типу `' OR 1=1 --` у параметр `q` призводить до зміни результатів запиту | Висока |
| Reflected XSS | При передачі `<script>alert(1)</script>` у параметр `msg` сторінка `/echo` виконує скрипт | Висока |
| Missing Security Headers | У відповідях відсутні заголовки `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options` | Середня |
| Cookie Flags | Session cookie не мають флагів `Secure` та `HttpOnly` | Середня |

Рекомендовано виправити ці проблеми: використовувати параметризовані SQL‑запити, екранувати HTML‑вихід, додати заголовки політик та флаги cookie.
